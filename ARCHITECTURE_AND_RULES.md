# Архитектура и Главные Правила Проекта

Этот документ описывает базовую архитектуру образовательной платформы `ProManas`, рассчитанной на высокие нагрузки (до 2000 одновременных пользователей, 1000 видеороликов, 5000 вопросов).

## 1. Общая архитектура
- **База данных:** Supabase (PostgreSQL). Отвечает за хранение пользователей, связей, вопросов, ответов, оценок и бизнес-логики.
- **Бэкенд (API):** Railway. Контейнерный хостинг. Отвечает за бизнес-логику, валидацию ответов, авторизацию и выдачу JSON-ответов фронтенду.
- **Фронтенд:** Разворачивается на Vercel или Cloudflare Pages (если это SPA). Оптимизированная раздача интерфейса пользователю.
- **Видео-контент:** Сторонний видео-хостинг с глобальным CDN (например, Cloudflare Stream, Mux или Vimeo).

## 2. Главные правила (Логика платформы)

### Логика видео и файлов
1. **Строгое разделение:** Railway занимается только API (отдает текст/JSON). Он **НИКОГДА** не раздает видеофайлы! Видео идет напрямую с видео-сервиса на клиентское устройство ученика.
2. **Адаптивный стриминг видео:** Никаких прямых `.mp4` файлов. Используется технология HLS или DASH (адаптивный битрейт), когда видео нарезается на мелкие чанки разного качества в зависимости от скорости интернета ученика. Видео загружается через админку в видео-сервис, а в Supabase сохраняется только строка-ссылка (ID) на HLS-поток.

### Логика базы данных и высоких нагрузок
3. **Пул соединений (Connection Pooling):** При работе бэкенда с Supabase обязательно используется конфигурация пула соединений — Connection Pooler (порт 6543), чтобы при одновременном тестировании тысяч учеников база данных не "упала" от избытка активных подключений.
4. **Горизонтальное масштабирование:** В Railway бэкенд настраивается с несколькими репликами (Replicas). Вместо одного сверхмощного сервера поднимаются 3-5 средних, между которыми распределяется нагрузка балансировщиком.

> Соблюдение этих правил гарантирует стабильную работу платформы под высокими нагрузками без зависаний, сбоев и перерасхода бюджета.
